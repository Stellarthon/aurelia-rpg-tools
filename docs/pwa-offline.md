# PWA / Offline App Shell

Makes the map **installable** (iPad "Add to Home Screen", desktop install) and
**loadable offline**. Pairs with the per-token content cache (`docs/per-player-
redaction-plan.md`): the **service worker** caches the *app shell* (`index.html`
+ icons), and the **content cache** holds the referee's authorised data — so a
full session survives a connectivity drop.

## Files
- `manifest.webmanifest` — install metadata (name, theme, icons). `start_url`/
  `scope` are relative (`.`) so it works whether the app is hosted at the root or
  a sub-path.
- `sw.js` — the service worker.
- `icons/icon-192.png`, `icon-512.png`, `apple-touch-180.png` — generated by
  `node tools/gen-icons.mjs` (a themed gold orrery ring). Replace with real art
  anytime — keep the same filenames/sizes, or re-run the generator.
- `index.html` — links the manifest + apple-touch-icon, sets `theme-color`, and
  registers the SW (secure-context only; failure is non-fatal).

## Hosting requirements
- **HTTPS** — the SW only registers on `https:` (it silently no-ops otherwise, so
  local `file://` previews still work, just without offline).
- All four files above must be served at the **same origin and path** as
  `index.html` (root is simplest). The SW's scope is its own directory.

## Update behaviour — important
`sw.js` is **network-first**: online, it always fetches the fresh `index.html`
and refreshes the cache, so **a republished build shows up immediately** — no
stale-cache trap. Offline, it serves the last cached shell. You normally never
need to touch the SW when you republish. If you ever want to force every client
to drop its cache, bump `CACHE = 'orion-shell-v1'` to `-v2` in `sw.js`.

## Verify
- **Desktop (Chrome/Edge):** DevTools → Application → Manifest (no errors) and
  Service Workers (activated). Toggle "Offline" and reload → app still loads.
- **iPad (Safari):** Share → Add to Home Screen → launch from the icon; it opens
  standalone (no browser chrome). Enable Airplane Mode and relaunch → the shell
  loads from cache, and referee content loads from the per-token cache.

## What the SW does NOT touch
Only same-origin `GET`s are intercepted. Supabase (`get-content`, storage globes)
and any API calls pass straight through — auth/dynamic data is never cached by
the SW (the content cache handles offline data separately, per token).
